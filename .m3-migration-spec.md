# M3 Migration Spec (Phase 3) — READ FULLY BEFORE EDITING

You are migrating raw Tailwind palette classes to native Material 3 token utility classes in a
Next.js + Tailwind v4 app. The M3 utility classes are defined in `src/app/globals.css`. A runtime
`!important` normalization layer currently maps raw classes to tokens — your job is to replace the
raw classes with the equivalent NATIVE token so the result is visually identical without relying on
that override. **Do not change any logic, props, data flow, layout structure, or text content.**
Only swap className color/border/background utilities and (where noted) add motion/elevation.

## HARD RULES
- Preserve all non-color classes exactly (spacing, flex, grid, sizing, typography sizes, etc.).
- Remove `dark:` variants of a raw class when you replace the base with a token that is already
  dark-mode aware (all `surface-*`, `on-surface*`, `outline*`, and `m3-*` tokens auto-adapt). E.g.
  `text-zinc-900 dark:text-white` becomes just `text-on-surface`.
- Keep opacity modifiers where meaningful: `border-outline-variant/50` is fine and preferred for
  borders (matches Phase 2 files). Card borders → `border-outline-variant/50`.
- Do NOT introduce new colors or restyle. Identical look is the goal.
- Do NOT touch files outside your assigned list.

`text-on-surface` only when the surface is neutral; if it sits on `bg-primary`/`bg-tertiary` etc.,
use the matching `text-on-*` token (`text-primary-foreground`, `text-on-tertiary`).

## SEMANTIC STATUS MAPPING
Success = emerald/green · Warning = amber/yellow · Error = red/rose · Info = blue · Accent = orange/sky
| Raw intent | text | solid bg | subtle bg | border | stroke (SVG) |
|---|---|---|---|---|---|
| Success (emerald/green) | `text-m3-success` | `bg-m3-success` | `bg-m3-success-subtle` | `border-m3-success` | `stroke-tertiary` |
| Warning (amber/yellow) | `text-m3-warning` | `bg-m3-warning` | `bg-m3-warning-subtle` | `border-m3-warning` | `stroke-primary` |
| Error (red/rose) | `text-m3-error` | `bg-m3-error` | `bg-m3-error-subtle` | `border-m3-error` | `stroke-error` |
| Info (blue) | `text-m3-info` | `bg-m3-info` | `bg-m3-info-subtle` | `border-m3-info` | — |
| Brand accent (orange, sky-as-accent) | `text-primary` | `bg-primary` (or `bg-primary/10`) | — | `border-primary` | `stroke-primary` |

- A `-500/10` or `-500/20` tinted background of a status color → the matching `bg-m3-*-subtle`.
- Solid destructive action buttons keep `bg-destructive text-white` (that shadcn token is fine).
- `sky` is ambiguous: if it denotes a link/brand/primary accent → `text-primary`/`bg-primary`. If it
  denotes an "info/in-progress" *status* (e.g. a status badge in a set with amber/emerald/red) →
  `text-m3-info`/`bg-m3-info-subtle`. Judge by surrounding sibling statuses.

## INTENTIONAL EXCEPTIONS — DO NOT MIGRATE THESE COLORS
- Calendar EVENT-CATEGORY colors: `indigo` (meetings), `cyan` (bookings/talent), `amber-400`
  holiday fill, fixed `today`/`holiday`/`weekend` day tints in `calendar-month-grid.tsx` and
  `leave/calendar/leave-calendar.tsx`. These are a fixed visual legend. ONLY migrate the neutral
  chrome (zinc backgrounds, borders, headings, muted text) in those files. Leave the colored event
  pills/legend swatches literal.
- Decorative brand/avatar gradients like `bg-gradient-to-br from-sky-400 to-orange-400`. Keep.
- If unsure whether a color is a fixed legend vs a status, PREFER leaving it literal and note it.

## STRUCTURE / MOTION (add while migrating, matching Phase 2 style)
- Top-level cards: add `elevation-1` and (if a rounded container) `shape-medium`/`shape-large`.
- Interactive cards / list rows that link or have hover: add `card-morph`.
- Icon action buttons: add `btn-morph`.
- List/grid of cards: add `morph-fade-in` to items and stagger with `morph-delay-1..8` where a small
  fixed number of siblings exists (don't stagger unbounded `.map()` beyond delay-8; it's fine to
  apply a single `morph-fade-in` to mapped items without per-item delay).
- Only ADD these; never remove existing animation classes. Don't add motion to tiny inline elements.

## PROCESS
1. Read the file fully.
2. Replace raw classes per the tables. Collapse now-redundant `dark:` pairs.
3. After editing, mentally verify no raw palette class remains EXCEPT allowed exceptions.
4. Report: file, count of replacements, and any colors you intentionally left literal (with reason).
